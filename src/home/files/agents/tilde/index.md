## ⚠️ Inbox session role — READ FIRST
- My inbox session's ONLY job is routing: read the update, find the chat where the work was planned (`chat list`/`chat show`), post a concise summary there, and stop. Do NOT investigate, debug, build, or fix anything in an inbox session — that's another session's job (babysitter/builder own it), and duplicating it burns tool calls for nothing (see #hmu1: I once spent 53 tool calls / ~12 min re-diagnosing a CI failure that babysitter was already fixing in parallel, then filed follow-up issues off my own investigation — @crawnk had to cancel it and filed #hmu1 about it).
- A few read-only lookups (issue show, gh pr checks/view, chat list/show) to confirm what already happened are fine and expected. But if the fix/investigation is already in-flight (comments from @builder/@babysitter show it's being handled), just route the update to chat and don't duplicate the work.
- Filing NEW follow-up issues from an inbox session should be rare — only when the update surfaces something nobody is already tracking. If in doubt, just route to chat; the chat's owner has the context to decide if new work is needed, the router doesn't.
- #hmu1 (open, owned by @crawnk) is a build-issue to fix this at the prompt level (waker.ts / orchestrator.ts). Once it lands, this warning may become redundant — check its status.
- **Useful trick for multi-update batches**: when several updates land in one inbox wake, don't just guess which issue each belongs to by title-matching — query the event log directly to be sure: write a throwaway script to `src/hull/_tmpquery.ts` (must live under src/hull for the `@hull/*` path alias to resolve) importing `systemDb`/`events` from `@hull/db/client`/`@hull/events/schema`, filter `like(events.type, 'issue.%')`, order by `desc(events.id)` limit ~15-40, and read the `topic` (`issue:<uuid>`) + `_notification.headline` fields — matches the update text exactly and the uuid maps 1:1 to an issue. Delete the tmp file after. `issue show <id>` only prints comments/status changes for ONE issue, so this is the fast way to disambiguate which issue a batch of headlines belongs to before spending calls on `issue show` per candidate.

## Issue system facts
- Issues carry no reference to chat (`originChatId` and `--chat` were removed — decouple-issues-from-chat). CLI: `npm run issue -- new|list|show|comment|handoff|playbooks|status|building|open|done|close` (run with SKYLARK_ACTOR=<my id>).
- An agent wake now lands on its own inbox session (find-or-create by well-known title, `findAgentSessionByTitle` in agent/service.ts), briefed on the whole unread batch. To route an update, use the new chat CLI: `npm run chat -- list|show <id>|post <id> <message>`.
- `npm run chat -- post <chatId> ...` requires the actor to be a member of that chat — use SKYLARK_ACTOR=<my id>.
- CLI arg parsing does NOT handle multi-line/backtick-heavy `--body` well when passed via `$(cat <<'EOF' ... EOF)` directly inline — angle brackets/backticks got eaten as shell redirection in one case even though quoted. Safer: write body to a tmp file first (`cat > /tmp/x.md <<'EOF' ... EOF`), inspect it, then pass `--body "$(cat /tmp/x.md)"`. If a title/body already got mangled, don't bother re-filing — just add a clean `issue comment <id> "$(cat /tmp/x.md)"` with the full text.
- `issue building <id>` kicks off the build playbook (@builder → @babysitter) immediately; multiple issues can build in parallel now (no longer "one build at a time" per the 2026-07-03 note — worth re-verifying gate status if things start colliding).
- When a build's PR shows red CI, don't just assume the code is broken — `gh pr checks <n>` / `gh run view <id> --log` (or `gh api .../actions/jobs/<id>/logs`) to read the actual failure before commenting. Sometimes it's infra (see npm version drift below), not the diff. BUT check issue comments first for whether babysitter/builder are already on it before duplicating investigation (see inbox role note above). After a fix lands, `gh pr checks <n>` takes several minutes to go green (~8min for coverage/verify) — poll with sleep rather than assuming instantly. Once merged, `issue done <id>` doesn't happen automatically — set it manually.

## Build/CI infra facts
- 2026-07-12: hit `npm ci` failing in CI (`EUSAGE ... Missing: lru-cache@x.y.z from lock file`) on an otherwise-trivial PR (#123, issue #3c5b). Root cause: nothing pins the *npm* binary version (only node via `.nvmrc`) — CI gets npm 10.9.8 (bundled with node 22.23.1 via actions/setup-node), but a builder's local/sandbox npm can be newer (e.g. 11.17.0) and produces a lockfile with subtly different optional-dep shape that npm 10.9.8's `npm ci` rejects as out-of-sync. Fix that worked: regenerate `package-lock.json` with `npx npm@10.9.8 install` (or whatever matches CI) before committing — landed on #3c5b/PR #123 (merged 2a9ae78, lockfile fix commit 027a1bb). Systemic follow-up **#iv1t** (pin npm via packageManager/corepack) building — check its status/PR before assuming future lockfile failures are code bugs. This was a one-off so far — #osy7/PR #124 built and merged clean with no CI issues, so it's not universal, just a risk to watch for.
