## Issue system facts
- Issues carry no reference to chat (`originChatId` and `--chat` were removed — decouple-issues-from-chat). CLI: `npm run issue -- new|list|show|comment|handoff|playbooks|status|building|open|done|close` (run with SKYLARK_ACTOR=<my id>).
- An agent wake now lands on its own inbox session (find-or-create by well-known title, `findAgentSessionByTitle` in agent/service.ts), briefed on the whole unread batch. To route an update, use the new chat CLI: `npm run chat -- list|show <id>|post <id> <message>`.
- `npm run chat -- post <chatId> ...` requires the actor to be a member of that chat — use SKYLARK_ACTOR=<my id>.
- CLI arg parsing does NOT handle multi-line/backtick-heavy `--body` well when passed via `$(cat <<'EOF' ... EOF)` directly inline in some shells — angle brackets/backticks got eaten as shell redirection in one case even though quoted. Safer: write body to a tmp file first, inspect it, then pass `--body "$(cat /tmp/x.md)"`. If a title/body already got mangled, don't bother re-filing — just add a clean `issue comment <id> "$(cat /tmp/x.md)"` with the full text.
- `issue building <id> --playbook build` kicks off the build playbook (@builder → @babysitter) immediately; multiple issues can build in parallel now (no longer "one build at a time" per the 2026-07-03 note).
- When a build's PR shows red CI, don't just assume the code is broken — `gh pr checks <n>`, `gh run view <id> --json headSha,headBranch,event`, `gh api /repos/<org>/<repo>/actions/jobs/<jobId>/logs` to read the actual failure before commenting. Sometimes it's infra (npm version drift, see below), not the diff.
- Each issue/branch typically has its own worktree at `/Users/crawnk/skylark/worktrees/<branch-name>/` — check `git worktree list` to find it rather than assuming your current cwd's node_modules apply to another branch's PR.
- Filing a near-duplicate follow-up issue for an immediate unblock is fine even if a systemic issue already exists for the root cause — just cross-link via `issue comment` (e.g. #59vb was the immediate PR-#123-lockfile fix, cross-linked to #iv1t which tracks the systemic npm-pinning fix). Babysitter/builder may resolve the immediate one directly on the PR branch and close the follow-up as moot — that's fine, check `issue show <id>` for the full thread before assuming it's still open.

## Build/CI infra facts
- 2026-07-12: hit `npm ci` failing in CI (`EUSAGE ... Missing: lru-cache@x.y.z from lock file`) on an otherwise-trivial PR (#123, issue #3c5b). Root cause: nothing pins the *npm* binary version (only node via `.nvmrc`) — CI gets npm 10.9.8 (bundled with node 22.23.1 via actions/setup-node), but a builder's local/sandbox npm can be newer (e.g. 11.17.0) and produces a lockfile with subtly different optional-dep shape (missing nitro's optional peer `node_modules/nitro/node_modules/lru-cache` entry, missing some `libc` fields, normalized bin paths) that npm 10.9.8's `npm ci` rejects as out-of-sync. Fix: regenerate `package-lock.json` with a clean `npm install` (ideally pinned npm version matching CI) before committing — don't hand-edit the lockfile.
  - **RESOLVED for #123**: babysitter regenerated the lockfile in-branch (commit 027a1bb) with npm 10.9.8, PR #123 merged as 2a9ae78, all checks green. #59vb (the follow-up I filed) closed as moot since the fix landed directly on the PR branch.
  - **#iv1t still open** — systemic fix (pin npm via packageManager/corepack) not yet built. Check its status before assuming future lockfile failures are code bugs; this WILL recur until #iv1t lands.
  - Local troubleshooting note: on this machine, npm's local cache (~/.npm/_cacache) is flaky/thrashy — `npm ci`/`npm install` intermittently fail with ENOTEMPTY/EEXIST/ENOENT tar errors unrelated to the actual lockfile issue. `rm -rf node_modules && npm ci` a second time (sometimes with `npm cache clean --force` first) usually clears it. Don't misdiagnose these as lockfile/code problems.
