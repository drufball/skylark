# tilde — memory index

## Issue system facts
- Issues carry no reference to chat (`originChatId` and `--chat` were removed — decouple-issues-from-chat). CLI: `npm run issue -- new|list|show|comment|handoff|playbooks|status|building|open|done|close` (run with SKYLARK_ACTOR=<my id>).
- An agent wake now lands on its own inbox session (find-or-create by well-known title, `findAgentSessionByTitle` in agent/service.ts), briefed on the whole unread batch. To route an update, use the new chat CLI: `npm run chat -- list|show <id>|post <id> <message>`.
- `npm run chat -- post <chatId> ...` requires the actor to be a member of that chat — use SKYLARK_ACTOR=<my id>.
- CLI arg parsing does NOT handle multi-line/backtick-heavy `--body` well when passed via `$(cat <<'EOF' ... EOF)` directly inline in some shells — angle brackets/backticks got eaten as shell redirection in one case even though quoted. Safer: write body to a tmp file first, inspect it, then pass `--body "$(cat /tmp/x.md)"`. If a title/body already got mangled, don't bother re-filing — just add a clean `issue comment <id> "$(cat /tmp/x.md)"` with the full text.
- `issue building <id> --playbook build` kicks off the build playbook (@builder → @babysitter) immediately; multiple issues can build in parallel now (no longer "one build at a time" per the 2026-07-03 note).
- When a build's PR shows red/no CI, don't just assume the code is broken — `gh pr checks <n>`, `gh pr view <n> --json mergeable,mergeStateStatus,statusCheckRollup`, `gh run view <id> --json headSha,headBranch,event`, `gh api /repos/<org>/<repo>/actions/jobs/<jobId>/logs` to read the actual failure/state before commenting. Sometimes it's infra (npm version drift) or a **stale-branch merge conflict from a sibling PR landing first** (zero CI checks + mergeStateStatus DIRTY/CONFLICTING is the tell for the latter — GitHub won't run checks on an unmergeable PR) — not the diff itself.
- Each issue/branch typically has its own worktree at `/Users/crawnk/skylark/worktrees/<branch-name>/` — check `git worktree list` to find it rather than assuming your current cwd's node_modules apply to another branch's PR.
- Filing a near-duplicate follow-up issue for an immediate unblock is fine even if a systemic issue already exists for the root cause — just cross-link via `issue comment` (e.g. #59vb was the immediate PR-#123-lockfile fix, cross-linked to #iv1t which tracks the systemic npm-pinning fix). Babysitter/builder may resolve the immediate one directly on the PR branch and close the follow-up as moot — check `issue show <id>` for the full thread before assuming it's still open.
- **Don't resolve merge conflicts yourself** — I inspect/diagnose (`git rebase origin/main` locally to see which files conflict, then `git rebase --abort` immediately) but leave the actual resolution to @builder/@babysitter via an `issue comment`. If a `git rebase` leaves stray state (e.g. rebase-merge dir persisting across worktree ops), `git rebase --abort` then `git checkout <branch> -f` to get back to the clean pushed state — verify with `git status`/`git log --oneline -3` before leaving it.

## Build/CI infra facts
- 2026-07-12: hit `npm ci` failing in CI (`EUSAGE ... Missing: lru-cache@x.y.z from lock file`) on PR #123 (issue #3c5b). Root cause: nothing pins the *npm* binary version (only node via `.nvmrc`) — CI gets npm 10.9.8, a builder's local npm can be newer and produces a subtly different lockfile shape that npm 10.9.8's `npm ci` rejects. **RESOLVED for #123** (babysitter regenerated lockfile in-branch, commit 027a1bb, PR merged as 2a9ae78). **#iv1t still open** — systemic fix (pin npm via packageManager/corepack) not yet built; this class of failure WILL recur until it lands.
  - Local npm cache (~/.npm/_cacache) on this machine is flaky — intermittent ENOTEMPTY/EEXIST/ENOENT tar errors unrelated to the actual lockfile issue. `rm -rf node_modules && npm ci` again (maybe `npm cache clean --force` first) usually clears it.
- 2026-07-12: PR #125 (issue #en2b) landed with zero CI checks + `mergeStateStatus: CONFLICTING` — branch was cut before sibling PR #124 merged, and both touched the same `<aside>` markup in chat.tsx/agent-chat.tsx/files.tsx (#124's CollapsibleSidebar refactor vs #125's overflow-hidden/min-h-0 additions). **General lesson: when multiple issues build in parallel against the same files, expect this** — check `gh pr view <n> --json mergeable,mergeStateStatus` early, not just `gh pr checks`, since a conflicting PR shows no checks at all (easy to misread as "still queued").
