- **Useful trick for multi-update batches / ambiguous headlines**: query the event log directly rather than guessing which issue an update belongs to: write a throwaway script to `src/hull/_tmpquery.ts` (must live under src/hull for the `@hull/*` path alias to resolve) importing `systemDb`/`events` from `@hull/db/client`/`@hull/events/schema`, filter `like(events.type, 'issue.%')`, order by `desc(events.id)` limit ~5-15, and read the `topic` (`issue:<uuid>`) + `_notification.headline` fields ‚Äî matches the update text exactly and the uuid maps 1:1 to an issue. Delete the tmp file after. `issue show <id>` only prints comments/status changes for ONE issue, so this is the fast way to disambiguate before spending calls on `issue show` per candidate.

## Issue system facts
- Issues carry no reference to chat (`originChatId` and `--chat` were removed ‚Äî decouple-issues-from-chat). CLI: `npm run issue -- new|list|show|comment|handoff|playbooks|status|building|open|done|close` (run with SKYLARK_ACTOR=<my id>).
- An agent wake now lands on its own inbox session (find-or-create by well-known title, `findAgentSessionByTitle` in agent/service.ts), briefed on the whole unread batch. To route an update, use the new chat CLI: `npm run chat -- list|show <id>|post <id> <message>`.
- `npm run chat -- post <chatId> ...` requires the actor to be a member of that chat ‚Äî use SKYLARK_ACTOR=<my id>.
- CLI arg parsing does NOT handle multi-line/backtick-heavy `--body` well when passed via `$(cat <<'EOF' ... EOF)` directly inline ‚Äî safer to write body to a tmp file first (`cat > /tmp/x.md <<'EOF' ... EOF`), inspect it, then pass `--body "$(cat /tmp/x.md)"`.
- `issue building <id>` kicks off the build playbook (@builder ‚Üí @babysitter) immediately; multiple issues can build in parallel now. Once merged, `issue done <id>` is often (not always) already set by babysitter ‚Äî check via `issue show` before setting it yourself (it'll error "Illegal issue transition: done ‚Üí done" harmlessly if already done, so no real risk either way).

## Build/CI infra facts
- 2026-07-12: hit `npm ci` failing in CI (`EUSAGE ... Missing: lru-cache@x.y.z from lock file`) on an otherwise-trivial PR (#123, issue #3c5b). Root cause: nothing pinned the *npm* binary version (only node via `.nvmrc`) ‚Äî CI gets npm 10.9.8 (bundled with node 22.23.1 via actions/setup-node), but a builder's local/sandbox npm could be newer (e.g. 11.17.0) and produce a lockfile with subtly different optional-dep shape that npm 10.9.8's `npm ci` rejects as out-of-sync.
- **Now permanently fixed**: #iv1t landed as PR #127 (merged c7414e3) ‚Äî `package.json` carries `"packageManager": "npm@10.9.8"`, and new `scripts/npm-cmd` auto-routes `scripts/setup`/`scripts/commit-gate` installs through `npx --yes corepack npm` whenever ambient npm differs from the pin (bare npm, zero overhead, when it already matches). This whole class of CI failure should no longer occur going forward ‚Äî if a lockfile/npm-ci failure crops up again after this, it's a genuinely new problem, not a recurrence of the old one.
